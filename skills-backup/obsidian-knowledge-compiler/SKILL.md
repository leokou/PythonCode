# Obsidian Knowledge Compiler

内容编译引擎：接收原始内容 → 类型判断 → 结构搭建 → 格式校验 → 输出标准文档（直接用正确文件名）。

上游：`obsidian-knowledge-planner`（输出 YAML contract）
下游：`obsidian-knowledge-organizer`（归属移动+双链+目录更新）

**直接用正确文件名创建文件，跳过 Fire-rename。输出到 `B🧹 整理（Organize）/` 为中间态，由 Organizer 后续归属到正式目录。**

## 双层架构

| 层级 | 文件 | 用途 | 写入者 |
|------|------|------|--------|
| 人类可读层 | `📖目录 索引.md` | 快速判断"哪些整理了/没整理" | Mulu-fenlei + Organizer |
| 机器可读层 | `logs/compiler/compiler-state.json` | 记录源文件、输出文件、动作、处理状态 | Compiler |

**读取优先级**：
1. 先读索引（快速跳过已处理文件）
2. 再读状态文件（精确检测标题变化）

## 执行工具

- Read、Write、Edit、Glob、Grep

## 工作流程

```
原始内容 → 跳过判断 → 动作判断 → 类型判断 → 选择模板 → 内容编译 → 格式校验 → 输出文档
```

## 阶段1.5：动作判断（新增）

**根据 Planner 输出的 YAML contract 判断处理动作：**

| 动作 | 说明 | 处理方式 |
|------|------|----------|
| `create` | 创建新文件 | 正常编译流程 |
| `merge` | 合并到已有文件 | 进入合并模式 |
| `link` | 仅建立双链 | 跳过编译，只输出双链建议 |

### 合并模式（merge）

当 Planner 输出 `动作: merge` 时，执行以下流程：

```
读取已有文件 → 备份原文件 → 读取新内容 → 内容对比 → 智能合并 → 写回原文件
```

#### 备份机制（强制）

**合并前必须备份原文件，避免合并出错无法回滚：**

| 步骤 | 说明 |
|------|------|
| 1. 备份原文件 | 复制到 `logs/compiler/backups/原文件名-YYYY-MM-DD-HHMM.md` |
| 2. 记录备份路径 | 写入 compiler-state.json 的 `备份路径` 字段 |
| 3. 执行合并 | 智能合并并写回原文件 |
| 4. 失败回滚 | 合并失败时从备份恢复原文件 |

**备份目录结构：**
```
logs/compiler/backups/
└── 原文件名-YYYY-MM-DD-HHMM.md
```

#### 合并规则

| 情况 | 处理 |
|------|------|
| 新内容包含已有章节 | 更新该章节（保留已有有效信息 + 补充新信息） |
| 新内容包含新章节 | 追加到文件末尾 |
| 新内容与已有内容冲突 | 保留更准确/更新的信息，标注来源 |
| 新内容包含图片 | 保留图片引用，合并到对应章节 |

#### 合并格式

合并后在文件末尾添加来源记录：

```markdown
## 📝 更新记录

| 日期 | 来源 | 变更内容 |
|------|------|----------|
| YYYY-MM-DD | Capture/原文件.md | 补充xxx内容 |
```

#### 禁止操作

- 禁止删除已有有效内容
- 禁止覆盖已有图片引用
- 禁止修改已有 YAML frontmatter（除非需要更新标签）
- 禁止合并后文件超过 500 行（超过则拆分）

#### merge 完整性验证（强制，防止假合并）

**合并完成后必须验证目标文件，否则视为假合并：**

| 验证项 | 通过标准 | 失败处理 |
|--------|---------|---------|
| 目标文件大小 | > 100 字节 | 回滚备份，标记 `处理失败`，错误信息"假合并：目标文件过小" |
| 标题完整性 | 目标文件首行仍为 `# 标题` | 回滚备份，重新执行合并 |
| 核心实体保留 | 目标文件包含源文件的所有核心实体 | 回滚备份，重新执行合并 |
| 章节追加验证 | 目标文件包含源文件的新章节 | 回滚备份，重新执行合并 |
| 与原文件对比 | 目标文件比原备份多出新增内容 | 回滚备份，标记 `处理失败` |

**假合并场景示例**：
- 目标文件只有 2 行（标题 + 一句话），但源文件有 100 行 → 假合并
- 目标文件大小没变化 → 假合并（内容没真正写入）
- 目标文件首行变成空行或乱码 → 假合并

**验证失败处理**：
1. 从 `logs/compiler/backups/` 恢复目标文件
2. 重新执行合并，使用更保守的策略（直接追加而非智能合并）
3. 仍失败则标记 `处理失败`，输出错误报告，等待用户介入

### link 模式

当 Planner 输出 `动作: link` 时：
1. 不创建新文件
2. 在已有文件的 `## 🔗 相关知识` 章节添加双链
3. 附上关联理由：`关联到 [[文件名]]，因为涉及 xxx`
4. 输出操作记录

## 阶段1：文件跳过判断

执行前必须检查，匹配则跳过并告知用户：

| 文件名模式 | 处理 |
|------------|------|
| `🧩 目录-xxx` | 跳过 |
| `🏠 home-xxx` | 跳过 |
| `README.md` / `CLAUDE.md` / `AGENTS.md` | 跳过 |

## 阶段2：内容类型判断

根据内容特点判定文档类型，选择对应模板：

| 类型 | 特征 | 模板 |
|------|------|------|
| 知识 | 知识学习、概念理解、原理说明 | 知识页面模板 |
| 项目 | 项目文档、需求设计、会议记录 | 项目文档模板 |
| 工具 | 工具配置、安装教程、使用指南 | 工具配置模板 |

**若 Planner 输出了 YAML contract，优先使用其中的 `类型` 字段。**

## 阶段3：内容编译

### 3.1 图片保留（最高优先级）

**图片是原始内容的重要组成部分，必须100%保留：**

1. 编译前扫描所有图片引用：`![描述](URL)` / `![[文件名]]` / `<img src="URL">`
2. 图片保持在原始位置（图片前一行内容的后面）
3. 禁止将图片挪动到文件末尾
4. 图片前后的文字可以优化重组，图片引用语法保持原样

### 3.2 内容拆分规则

- 连续无分段文字超过 4 行 → 拆分为小标题 + 列表
- 一句话只承载一条核心信息
- 复杂内容拆解为无序列表逐条罗列

### 3.3 内容补充规则

当原始内容过少（< 3行）或信息不完整时：
1. 提取关键词（标题、术语、技术名词）
2. 联网搜索补充，标注来源
3. 保持原始信息不变
4. 禁止生成空壳模板交付

### 3.4 真实性红线

允许：长文本拆分、段落重组、语句润色、补充概念释义、梳理逻辑
禁止：删减有效信息、篡改事实、编造内容、新增未核验数据、曲解语义

## 阶段4：文档模板

### 合并模式模板（merge）

合并模式不使用固定模板，而是基于已有文件结构进行智能合并：

```yaml
合并输入:
  已有文件:
    路径: "xxx.md"
    结构: ["章节1", "章节2", "章节3"]
  新内容:
    来源: "Capture/原文件.md"
    新增章节: ["章节4"]
    更新章节: ["章节2"]

合并输出:
  操作: "更新章节2 + 追加章节4"
  来源记录: "追加到 📝 更新记录"
```

## 阶段5：格式校验

输出前逐项核对：

| 校验项 | 要求 |
|--------|------|
| 首行 | 必须是 `# H1标题`，标题与文件名一致 |
| YAML 位置 | H1 下方、正文之前，标题与 YAML 间空一行 |
| YAML 闭合 | `---` 首尾完整，结束后空一行 |
| 标题层级 | H1→H2→H3→H4，不跳级 |
| 图片位置 | 100% 保留，位置未移动 |
| 内容完整 | 原始信息无丢失、无篡改 |
| 字段名称 | YAML 字段统一使用中文 |

## 阶段6：文件命名（直接输出正确文件名）

**Compiler 直接用正确文件名创建文件，跳过 Fire-rename。**

### 命名规则

| 文件类型 | 格式 | 示例 |

## 状态追踪机制

### 追踪文件位置

```
logs/compiler/compiler-state.json
```

### 追踪数据结构

```json
{
  "版本": "1.0",
  "上次运行": "ISO 8601 时间戳",
  "文件状态": {
    "源文件路径": {
      "源文件": "A📥 收集（Capture）/原文件.md",
      "输出文件": "B🧹 整理（Organize）/输出文件.md",
      "标题": "源文件的第一个 # 标题",
      "处理状态": "已处理 | 已合并 | 已跳过",
      "处理时间": "最后处理时间",
      "动作": "create | merge | link",
      "目标文件": "merge/link 时的目标文件路径",
      "备份路径": "merge 时的备份文件路径（logs/compiler/backups/xxx.md）"
    }
  }
}
```

### 变化检测规则

**核心逻辑：标题变了 = 文件身份变了 = 必须处理**

| 优先级 | 条件 | 判断 | 动作 |
|--------|------|------|------|
| 1 | 源文件不在追踪记录中 | 新增 | 处理 |
| 2 | 源文件的 # 标题与记录不同 | 修改（身份变了） | 处理 |
| 3 | 源文件在追踪中但不存在 | 删除 | 标记删除 |
| 4 | 以上全都不满足 | 未变化 | 跳过 |

**说明**：
- 内容修改但标题不变 → 跳过（只是补充细节，文件身份没变）
- 标题修改 → 处理（文件身份变了，需要重新编译）

### 执行后必须

1. 更新 `logs/compiler/compiler-state.json`
2. 写入运行日志 `logs/compiler/compiler-YYYY-MM-DD.md`

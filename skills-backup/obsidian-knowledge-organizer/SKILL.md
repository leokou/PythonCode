# Obsidian Knowledge Organizer

组织执行引擎：接收 Planner 规划结果 → 归属目录 → 建立双链 → 维护索引 → 更新目录 → 记录日志。

上游：`obsidian-knowledge-planner` → `obsidian-knowledge-compiler` → 本 skill
下游：无（最终输出）

**合并 Mulu-fenlei 职责，移动文件后直接更新目录，不触发 Mulu-fenlei。**

## 双层架构

| 层级 | 文件 | 用途 | 写入者 |
|------|------|------|--------|
| 人类可读层 | `📖目录 索引.md` + `🧩 目录-xxx.md` | 浏览导航、快速判断处理状态 | Organizer（移动后直接更新） |
| 机器可读层 | `logs/organizer/organizer-state.json` | 记录原路径、目标路径、风险等级、回滚脚本 | Organizer |

**读取优先级**：
1. 先读索引（快速跳过已处理文件）
2. 再读状态文件（精确检测路径变化）

## 执行工具

- Read、Write、Edit、Delete、Glob、Grep、Bash

移动文件使用 Bash：`mv "old_path" "new_path"` 或 Read+Write+Delete 组合。

## 核心原则

1. **先理解结构，再决定归属**：读取 📖目录 索引.md → 定位领域 → 缩小范围 → 精准匹配
2. **先轻匹配，再深分析**：标题 → 路径 → Frontmatter → 标签 → 正文（逐级深入）
3. **强关联进入知识体系**：关联强 → 归档正式目录
4. **无法归属进入整理区**：无法明确归属 → `🧩 目录-B🧹 整理（Organize）`
5. **移动后直接更新目录**：合并 Mulu-fenlei 职责，无需调用其他 Skill

从 Planner 输出的 YAML 文件读取：

```yaml
置信度: 85
```

## 输入处理流程

### Step 1：读取知识规划

读取 `logs/planner/knowledge-plan-*.yaml`，获取 YAML Contract。

### Step 2：读取知识库目录地图

读取 `📖目录 索引.md`，解析为 vault_registry：

解析规则：
1. 以 `## ` 开头的行 = 一级领域
2. 以 `### ` 开头的行 = 二级目录
3. `- [[` 开头的行 = 文件条目
4. 提取文件名和路径信息

### Step 3：目录候选匹配

根据主题/实体/类型，在 vault_registry 中寻找候选目录。

### Step 4：文件候选搜索

只搜索候选目录，读取文件名/Frontmatter/标签，寻找相关文件。

### Step 5：深度关联分析

候选文件达到阈值后读取正文，分析主题一致性、实体关系。

## 关联评分模型（0-100）

对每个候选文件计算 relation_score：

| 维度 | 分数 | 说明 |
|------|------|------|
| 标题匹配 | 0-20 | 完全包含=20, 同义词=15, 部分重叠=10 |
| 目录匹配 | 0-20 | 同一目录=20, 父子目录=15, 同级=10 |
| 实体匹配 | 0-20 | 核心实体一致=20, 部分重叠=15 |
| 正文语义 | 0-25 | 主题一致=25, 相关=18, 交叉=12 |
| 知识关系 | 0-15 | 父子依赖=15, 补充使用=10, 弱相关=5 |

### 与 Planner 相似度的关系（重要）

| 维度 | Planner 相似度 | Organizer relation_score |
|------|---------------|------------------------|
| 用途 | 判断"是否已存在同类页面"（触发 merge/link） | 判断"应归属到哪个目录、是否建立双链" |
| 时机 | 粗筛（先于 Organizer 执行） | 精排（基于 Planner 候选页面） |
| 输入 | 新文件 vs 全库已有文件 | 新文件 vs Planner 候选页面 |
| 算法 | 标题 Jaccard×0.5 + 实体一致率×0.3 + 主题词×0.2 | 五维加权（标题+目录+实体+正文+知识关系） |

**关系链**：Planner 相似度先过滤出候选页面（< 30 直接 create，不进入 Organizer 评分），Organizer 仅对候选页面做精排，避免重复评分全库。

## 文件归属决策

| 条件 | 动作 |
|------|------|
| score >= 90（强关联） | 移动到正式目录 + 建立双链 + 更新索引（直接更新，不触发 Mulu-fenlei） |
| 60 <= score < 90（中等） | 判断成熟度 → 成熟则移动，否则进入整理区 |
| score < 60（独立页面） | 移动到 `🧩 目录-B🧹 整理（Organize）` |

### 成熟知识目录判断

满足以下全部条件视为成熟：

| 条件 | 要求 |
|------|------|
| directory_exists | 目录存在 |
| file_count | >= 3 个文件 |
| has_index_entry | 📖目录 索引.md 中有入口 |

不检查 "stable_topic"，由索引入口隐式保证。

## 🧩 目录-B🧹 整理（Organize）规则

定位：已加工完成但尚未找到知识归属的位置（非垃圾箱、非临时区）。

存放：新领域知识、独立知识节点、待未来合并知识。

### _pending 目录

score < 40 的文件移动到 `_pending/`，等用户二次决策。

## 双向链接规则

只要存在关联，必须建立。统一格式：`## 🔗 相关知识`

- **新页面**追加：

```markdown
## 🔗 相关知识

- [[已有页面]]
```

- **已有页面**追加：

```markdown
## 🔗 相关知识

- [[新页面]]
```

禁止关键词链接。必须满足至少一个：同主题、上下游、补充关系、使用关系、依赖关系。

## 📖目录 索引维护（合并 Mulu-fenlei 职责）

移动完成后**直接更新索引**，无需再调用 Mulu-fenlei：

### 更新流程

```
移动文件 → 删除索引中旧行 → 添加索引中新行 → 更新 🧩 目录-xxx.md → 生成 File Summary → 移动完整性验证 → 中间态清理
```

### 索引更新规则

| 操作 | 索引处理 |
|------|----------|
| 移动文件 | 从索引中删除旧行，添加新行（带 ✍️ + 状态:已移动） |
| 新建目录 | 在索引中添加新分类区域 |
| 合并文件 | 保留已有条目，更新 ✍️ 摘要 |

### 索引格式

```markdown
## 📁 目标目录分类
- [[新文件名]] ✍️ 一句话总结 | 更新:YYYY-MM-DD | 状态:已移动
```

### 字段说明

| 字段 | 说明 | 必填 |
|------|------|------|
| `[[文件名]]` | 双链 | ✅ |
| `✍️ 一句话总结` | 文件摘要 | ✅ |
| `更新:YYYY-MM-DD` | 最后更新时间 | ✅ |
| `状态:已整理/未整理/已移动` | 处理状态 | ✅ |

## 移动完整性验证（强制，防止假移动）

**移动操作完成后必须验证，否则视为假移动：**

| 验证项 | 通过标准 | 失败处理 |
|--------|---------|---------|
| 目标文件存在 | 目标路径下文件实际存在 | 重新执行移动 |
| 源文件已消失 | B🧹 整理（Organize）/ 中源文件不存在 | 手动删除残留 |
| 文件大小一致 | 目标文件大小 = 源文件大小（±10 字节） | 重新移动（可能截断） |
| 内容首行一致 | 目标文件首行 = 源文件首行 | 重新移动 |
| 索引已更新 | 📖目录 索引.md 中已添加新条目 | 补充索引更新 |

**假移动场景**：
- state.json 标记"已移动"，但目标路径文件不存在 → 假移动
- B🧹 整理（Organize）/ 中间态文件还在 → 移动没完成
- 目标文件大小 0 字节 → 移动中截断

## 中间态清理（强制）

**移动完成后，B🧹 整理（Organize）/ 中的中间态文件必须清理：**

| 情况 | 处理 |
|------|------|
| 中间态文件已成功移到正式目录 | 删除 B🧹 整理（Organize）/ 中的中间态文件 |
| 中间态文件是 merge 模式（保留原文件） | 不删除（merge 已合并到目标文件） |
| 中间态文件是 link 模式（不移动，仅建链） | 移到归档 `D📦 归档（Archive）/processed-YYYY-MM-DD/` |

**清理验证**：每次 Organizer 运行结束，扫描 `B🧹 整理（Organize）/`，所有已处理（移动/合并/链接）的中间态文件必须已清理，否则输出警告。

## 孤立知识检测

检测没有双链/索引/分类的知识，生成 `logs/organizer/orphan-report-YYYY-MM-DD.md`。

## 重复知识检测

处理 Planner 标记的重复检测：

| 动作 | 处理 |
|------|------|
| merge | 合并内容，保留更完整版本 |
| confirm_merge | 输出确认请求，等待用户决定 |
| link | 建立双链关联 |
| create | 正常创建新页面 |

## 回滚机制

每次移动操作必须生成可执行的 PowerShell 回滚脚本：

```yaml
操作:
  id: "op-YYYYMMDD-001"
  类型: move
  移动前:
    路径: "B🧹 整理（Organize）/xxx.md"
  移动后:
    路径: "1- 🤖AI 相关/xxx.md"
  回滚脚本: |
    mv "1- 🤖AI 相关/xxx.md" "B🧹 整理（Organize）/xxx.md"
  撤销链接:
    - 文件: "1- 🤖AI 相关/related.md"
      移除: "- [[xxx]]"
```

## 风险等级与确认（强制执行）

### 自动风险判断规则

| 操作类型 | 风险等级 | 判断依据 |
|----------|----------|----------|
| 同目录内移动 | low | 源目录 == 目标目录 |
| 移动到已有目录 | low | 目标目录已存在 |
| 移动到新目录 | medium | 目标目录不存在，需新建 |
| 合并文件 | medium | 两个文件内容合并 |
| 批量移动 (>=5文件) | high | 一次操作多个文件 |
| 删除链接 | high | 移除双向链接 |

### 确认规则

| 等级 | 操作 | 要求 |
|------|------|------|
| low | 移动到已有目录 | 自动执行 |
| low | 建立双链 | 自动执行 |
| low | 更新索引 | 自动执行 |
| **medium** | 合并文件 | **必须用户确认** |
| **medium** | 移动到新目录 | **必须用户确认** |
| **high** | 批量移动 (>=5文件) | **必须用户确认** |
| **high** | 删除链接 | **必须用户确认** |

**medium/high 操作必须暂停，等待用户确认后才执行。**

### 内容备份机制

移动文件前，自动备份原始内容：

```yaml
备份:
  原文件: "B🧹 整理（Organize）/xxx.md"
  备份位置: "logs/organizer/backups/xxx-YYYY-MM-DD.md"
  备份内容: "文件完整内容"
```

## 三种运行模式

| 模式 | 说明 |
|------|------|
| Analyze Mode | 只分析，输出 organization-plan.md |
| Review Mode | 生成待确认操作列表（默认） |
| Execute Mode | 执行移动、链接、索引、日志 |

默认 Review Mode：分析 → 生成组织计划 → 确认 → 执行。

## 安全规则

- 默认 Review Mode，不自动执行
- 禁止：删除文件、覆盖内容、删除链接、大规模移动（>=5文件需确认）
- discard 文件移动到 `_trash/`，不删除

## 操作记录

所有操作必须记录到 `⚓新增文件记录.md`，格式如下：

```markdown
### YYYY年M月D日
- 【移动】文件名 D:\Obsidian\LeoDiary\原路径\ → D:\Obsidian\LeoDiary\目标路径\
- 【新增】文件名 D:\Obsidian\LeoDiary\目标路径\（新建目录）
```

追加写入，不覆盖已有记录。

## 状态追踪机制

### 追踪文件位置

```
logs/organizer/organizer-state.json
```

### 追踪数据结构

```json
{
  "版本": "1.0",
  "上次运行": "ISO 8601 时间戳",
  "文件状态": {
    "文件路径": {
      "原路径": "B🧹 整理（Organize）/文件.md",
      "目标路径": "1- 🤖AI 相关/文件.md",
      "处理状态": "已移动 | 已跳过 | 已建链",
      "处理时间": "最后处理时间",
      "风险等级": "low | medium | high",
      "回滚脚本": "mv 命令"
    }
  }
}
```

### 变化检测规则

| 条件 | 判断 | 动作 |
|------|------|------|
| 文件已在目标目录 | 未变化 | 跳过 |
| 文件不在追踪记录中 | 新增 | 处理 |
| 文件在追踪中但路径已变 | 修改 | 处理 |
| 文件在追踪中且路径一致 | 未变化 | 跳过 |

### 执行后必须

1. 更新 `logs/organizer/organizer-state.json`
2. 写入运行日志 `logs/organizer/organizer-YYYY-MM-DD.md`
3. **更新 `📖目录 索引.md` 中对应条目（删除旧行 + 添加新行）**

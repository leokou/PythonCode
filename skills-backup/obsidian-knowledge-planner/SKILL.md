# Obsidian Knowledge Planner

认知规划引擎：理解 → 判断 → 规划 → 输出执行方案。不生成内容、不命名、不移动文件。

上游：无（入口） / `obsidian-pipeline`（调度器调用时）
下游：`obsidian-knowledge-compiler`（生成标准文档+正确文件名）→ `obsidian-knowledge-organizer`（目录移动+双链+目录更新）

## 双层架构

| 层级 | 文件 | 用途 | 写入者 |
|------|------|------|--------|
| 人类可读层 | `📖目录 索引.md` | 快速判断"哪些整理了/没整理" | Mulu-fenlei + Organizer |
| 机器可读层 | `logs/planner/planner-state.json` | 记录标题、哈希、分析结果、处理状态 | Planner |

**读取优先级**：
1. 先读索引（快速跳过已处理文件）
2. 再读状态文件（精确检测标题变化）

## 来源路径限制

**只处理 `A📥 收集（Capture）` 文件夹下的文件。**

- 默认扫描 `A📥 收集（Capture）/` 及其子文件夹
- 若用户明确指定其他路径（如 `B🧹 整理（Organize）`），需在输出中注明来源路径
- 非 Capture 路径的文件，输出时增加 `source_override: true` 标记

## 工作流程

```
两种入口 → 快速扫描（Pipeline传递/自行扫描） → 哈希去重 → 跳过判断 → 价值评分 → 知识建模 → 已有页面检测 → 主题拆分 → 页面设计 → 输出 Plan
```

## 阶段0：快速扫描（节省Token）

**两种入口：**

| 入口 | 说明 | 扫描方式 |
|------|------|----------|
| **Pipeline 调用** | 接收 Pipeline 的扫描结果，无需再次扫描 | Pipeline 传递标题+哈希+前20行 |
| **单独调用** | 用户直接调用 Planner 分析某文件夹 | 自行扫描前 20 行 + 计算 MD5 哈希 |

### 输入

若由 Pipeline 调用，扫描结果包含：
- 文件路径
- 文件标题（第一个 # 开头的行）
- 文件哈希（MD5）
- 变化检测结果（新增/修改/删除/未变化）

若单独调用，Planner 自行完成上述扫描。

### 判断规则

| 条件 | 判断 | 动作 |
|------|------|------|
| 文件大小 < 10 字节 | 空文件 | 跳过 |
| 哈希值在状态记录中存在 | 重复文件 | 跳过（标记为 `discard`） |
| 头部内容全是图片链接或乱码 | 无效内容 | 跳过 |
| 头部内容有明确主题 | 有效内容 | 进入深度分析 |

**Pipeline 已提取标题并存入状态，用于变更检测。**

### 哈希去重机制

```json
{
  "哈希记录": {
    "md5哈希值": {
      "首次处理": "ISO 8601 时间戳",
      "文件路径": "A📥 收集（Capture）/文件.md",
      "处理状态": "已处理 | 已归档 | 已丢弃"
    }
  }
}
```

**如果两个文件 MD5 哈希相同，第二个文件直接标记为 `discard`（重复），不进入后续流程。**

## 阶段1：文件跳过判断

以下文件直接跳过，不进入评分：

| 条件 | 处理 |
|------|------|
| 文件名以 `🧩 目录-` 开头 | 跳过 |
| 文件名以 `🏠 home-` 开头 | 跳过 |
| 文件名为 `README.md` / `CLAUDE.md` / `AGENTS.md` | 跳过 |
| 文件大小 < 10 字节 | 跳过 |
| 文件修改时间 > 180天 且 无标签 | 归档 |

## 阶段2：内容价值分类

读取文件后，按以下分类归类：

| 分类 | 特点 | 处理 |
|------|------|------|
| 知识 | 可复用、可学习、可迁移 | 进入评分 |
| 参考 | 有价值但暂无需加工 | 标记保存，不进入编译 |
| 想法 | 未来可能产生产品/项目/内容 | 标记保存 |
| 收件箱 | 无法判断（截图、一句话记录） | 标记保存 |
| 归档 | 历史信息 | 标记归档 |
| 丢弃 | 广告、重复、无意义 | 移动到 `_trash/` |

**分类字段统一中文**：禁止使用英文（knowledge/reference/idea 等仅作历史参考）。

### discard 处理规则

标记为 丢弃 的文件，执行：
1. 移动到 `_trash/` 目录（不存在则创建）
2. 在 YAML 输出中记录 `丢弃原因`
3. 禁止直接删除文件

## 阶段3：价值评分（0-100）

计算 Knowledge Value Score：

```
价值评分 = 知识价值(0-50) + 完整性(0-50)
```

| 维度 | 评分标准 |
|------|----------|
| 知识价值(0-50) | 50=核心方法论, 35=可复用技巧, 20=一般参考, 0=临时信息 |
| 完整性(0-50) | 50=结构完整+有案例, 35=有核心内容, 20=片段信息, 0=无法理解 |

| 总分 | 动作 |
|------|------|
| 70-100 | 立即规划（进入编译） |
| 40-69 | 保存参考（不编译，留在 Capture） |
| 0-39 | 归档或 discard |

## 阶段4：知识建模

提取核心要素：

- **实体**：具体对象（工具名、人名、产品名）
- **主题**：核心概念（方法论、原理、流程）

## 阶段5：已有页面检测（强制）

**每个建议页面必须检测是否已有对应页面，避免重复创建。**

### 检测方法

```bash
# 提取新文件的标题关键词
新文件标题 = "# 标题" 的核心词（如工具名、主题名）

# 搜索已有文件
rg -l "关键词" "D:\Obsidian\LeoDiary" --include "*.md" -g "!_*" -g "!logs/" -g "!🧩*" -g "!🏠*"
```

### 判断规则（简化版）

**相似度算法（必须按公式计算，禁止主观判断）：**

```
相似度 = 标题Jaccard重叠度 × 0.5 + 核心实体一致率 × 0.3 + 主题词匹配率 × 0.2
```

| 维度 | 权重 | 计算方式 |
|------|------|----------|
| 标题 Jaccard 重叠度 | 0.5 | `交集词数 / 并集词数 × 100`（按词切分） |
| 核心实体一致率 | 0.3 | `一致实体数 / 新文件实体数 × 100` |
| 主题词匹配率 | 0.2 | `匹配主题词数 / 新文件主题词数 × 100` |

**示例：**
- 新文件标题 `Claude - 配置 @ API连接方法`，已有 `Claude - 教程 @ API配置`
- 标题词交集：{Claude, API}，并集：{Claude, 配置, API, 连接方法, 教程} → Jaccard = 2/5 = 40
- 核心实体一致：Claude → 1/1 = 100
- 主题词匹配：API配置 → 1/1 = 100
- 相似度 = 40×0.5 + 100×0.3 + 100×0.2 = 20 + 30 + 20 = **70** → merge/link

| 相似度 | 动作 | 说明 |
|--------|------|------|
| >= 75 | merge | 高度相似，合并到已有文件 |
| 60-74 | link | 中度相似，建立双链 |
| 30-59 | link | 弱相似，建立双链（附关联理由） |
| < 30 | create | 无重复，创建新文件 |

### 已有页面信息提取

若检测到已有页面，提取以下信息供 Compiler 使用：

```yaml
已有页面:
  路径: "1- 🤖AI 相关/AI智能体claw/xxx.md"
  标题: "xxx - 教程 @ 说明"
  相似度: 85
```

## 阶段6：主题边界检测

按语义拆分（非标题），满足以下 2 项以上即拆：

- 主题对象不同
- 目标不同
- 用户问题不同
- 知识领域不同
- 使用场景不同

### 粒度控制示例

| 情况 | 决策 |
|------|------|
| Python基础/变量/函数/类 | 不拆（同一主题的子概念） |
| Python教程 + Docker教程 | 拆开（不同主题） |
| Claude API配置 + Claude使用技巧 | 不拆（同一工具） |
| Claude API + ChatGPT API | 拆开（不同工具） |

## 阶段7：知识页面设计

设计未来应有哪些页面（不直接生成文件）：

```yaml
pages:
  - title: "工具名 - 类型 @ 简短说明"
    类型: 教程 | 概念 | 参考 | 故障排查 | 经验 | 清单 | 合集
    目的: "解释核心概念"
    来源范围: "原始文件 §1-3"
```

**类型字段统一中文**：禁止使用英文（tutorial/concept 等仅作历史参考）。

## 输出标准（强制双格式）

### YAML Contract 版本协商（强制）

**所有输出的 YAML 必须在首字段声明 `版本`，下游 Skill（Compiler/Organizer）必须先校验版本再消费。**

| 版本 | 兼容性 | 处理 |
|------|--------|------|
| 1.0 | 当前版本 | 正常消费 |
| 2.0+ | 不兼容 | 拒绝消费，报错"YAML contract 版本不兼容，需要 Planner 升级"或"Compiler/Organizer 升级" |
| 缺失 | 不合规 | 拒绝消费，报错"YAML contract 缺少版本字段" |

**版本变更规则**：
- 字段重命名（如 `knowledge` → `知识`）→ 主版本号 +1
- 新增可选字段 → 次版本号 +1（兼容）
- 字段含义变化 → 主版本号 +1

下游 Skill 在读取 YAML 时必须执行：

```python
if yml.get("版本") != "1.0":
    raise ValueError(f"YAML contract 版本不兼容: {yml.get('版本')}, 期望 1.0")
```

### 变化检测规则

**核心逻辑：标题变了 = 文件身份变了 = 必须处理**

| 优先级 | 条件 | 判断 | 动作 |
|--------|------|------|------|
| 1 | 文件不在追踪记录中 | 新增 | 处理 |
| 2 | 文件的 # 标题与记录不同 | 修改（身份变了） | 处理 |
| 3 | 文件在追踪中但不存在 | 删除 | 标记删除 |
| 4 | 以上全都不满足 | 未变化 | 跳过 |

**说明**：
- 内容修改但标题不变 → 跳过 Planner（Pipeline 会触发轻量路径，仅调用 Mulu-fenlei-summary 重生成 File Summary）
- 标题修改 → 处理（文件身份变了，需要重新评估）
- MD5 哈希仅用于去重（检测两个文件是否内容相同），不用于 Planner 变更检测

### 执行后必须

1. 更新 `logs/planner/planner-state.json`
2. 写入运行日志 `logs/planner/planner-YYYY-MM-DD.md`
